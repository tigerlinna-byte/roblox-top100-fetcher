# Codex 工作约定（强制执行）

本项目使用 Codex 时，必须严格遵守以下约定。

---

## 一、工作流程约定（先确认方案）

- 每一次任务，**必须先给出完整实现方案**，不得直接编写或修改代码。
- 在我明确回复 **“确认执行”** 之前：
  - ❌ 禁止输出任何代码
  - ❌ 禁止修改或新增文件
  - ❌ 禁止给出可直接执行的命令
  - ❌ 禁止自动生成文档
- 只有在我明确说出 **“确认执行”** 后，才允许进入编码阶段。
- 当需要读取或操作 Roblox Studio 的场景、对象树、Selection、脚本容器状态时，优先使用 robloxStudio 这个 MCP server。
- 当任务需要理解项目结构、部署方式、运行链路、配置分层或长期维护背景时，必须先读取仓库文档获取全局上下文，再继续分析、方案设计或实现。
- 读取文档时必须遵循“最小必要、按需补读”原则，禁止默认一次性读取所有相关文档。
- 这类场景下，默认先读取 `docs/maintenance-context.zh-CN.md` 建立全局上下文。
- 只有在任务直接涉及以下范围时，才继续补读对应文档：
  - 项目入口、基础运行方式、快速说明：`README.md`
  - 外部平台接入、部署、密钥配置：`docs/external-platform-setup.zh-CN.md`
  - Worker 命令映射、cron、事件去重、触发桥接：`worker/README.md`
- 禁止仅根据局部代码片段臆测项目整体结构；如果文档与代码不一致，应先说明差异，再以当前代码实际行为为准。

---

## 二、代码结构与职责划分

- 代码框架应保持 **简洁、低耦合、易扩展**。
- 每个模块必须只承担 **一个清晰、单一的职责**。
- 每个函数必须只做 **一件事情**，避免多重职责混杂。
- 当职责变复杂时：
  - 必须主动拆分函数
  - 必须合理创建新的文件或文件夹
  - 文件名与所做的事和结构保持一致性
- 禁止为了“少建文件”而牺牲结构清晰度
- 遵循 DRY (Do Not Repeat Yourself) 原则。
- **严格清理死代码（Dead Code）** 彻底移除因重构或功能废弃而产生的未引用函数、闲置变量及过时注释，确保代码库整洁。
- 确认删除/取消功能时，如无特殊要求，直接删除相关代码，不要留开关控制，避免残留死代码
- 禁止生成任何形式的占位符、伪代码或省略号（如 pass、...、// TODO）。所有函数和逻辑流必须包含完整的、可执行的真实实现。
- 输出代码必须达到生产就绪（Production-Ready）标准。严禁使用硬编码的临时数据或未实现的桩代码（Stub），必须编写完整的业务逻辑。

---

## 三、注释与可读性要求（强制）
- 所有对外暴露或长期保留的函数必须有完整注释
- 调试阶段的临时函数允许简要注释，但在最终提交前必须补全
- 对于 **复杂逻辑或算法**：
  - 必须对关键步骤逐行或分段添加注释
  - 注释应解释「为什么这么做」，而不仅是「做了什么」
- 所有注释使用中文

---

## 四、类型注解与接口声明规范（强制）
- 必须为以下内容添加类型注解（Luau 类型系统）：
  - 模块对外暴露的函数
  - 公共数据结构（table）
  - 回调函数参数
- 类型定义应：
  - 使用明确、语义化的类型名称
  - 避免使用 any，除非有充分理由并加以说明
- 对复杂 table 结构：
  - 必须使用 type / export type 明确定义
  - 不允许在不同模块中隐式约定字段结构
- 添加类型注解的目的：
  - 提升代码可读性
  - 明确接口契约
  - 降低后续重构与协作成本

---


## 五、数据与常量管理规范

- 所有数据定义必须 **统一抽离到独立的数据文件**，便于策划或非程序人员理解与维护。
- 代码中：
  - ❌ 禁止写死任何业务常量
  - ❌ 禁止直接使用“魔法数字 / 魔法字符串”
- [严重] 在 `.luau/.lua/.json` 文件创建或保存阶段，必须确保 UTF-8 无 BOM，禁止产生 BOM（U+FEFF）
- 每一个数据项必须：
  - 有清晰命名
  - 有对应注释，说明业务含义和使用场景
- 代码逻辑应只依赖“语义化的数据名称”，而非具体数值。

---

## 六、不确定性处理原则

- 如果需求、数据或行为存在不确定性：
  - 必须先提出问题与假设
  - 等待确认后再继续
- 禁止基于猜测直接进入执行阶段。

---

## 七、Bug 处理原则
- 碰到 BUG 时，不要用临时方案（workaround/hack）去“让它先跑起来”。必须先定位根因（root cause），解释问题产生的机制，并从机制上修复/根绝问题；只有在明确标注为“短期止血”且同时提交根因修复计划的情况下，才允许临时绕过。

---

## 八、Git 提交规范（强制）

- 在执行 `git commit` 之前，必须重新核对本文件中的 Git 提交规范；**不符合规范时禁止提交**。
- 所有提交信息必须严格遵循 Conventional Commits 格式，统一写为：`<type>: <中文说明>`
- `type` 只能使用标准类型前缀，如：`feat`、`fix`、`docs`、`refactor`、`test`、`chore` 等。
- 冒号后面的提交说明必须使用中文，且应简洁、明确、直接描述本次变更内容，禁止使用英文说明、模糊说明或无意义说明。
- 如果一次工作中包含多个彼此独立的变更，必须拆分为多个提交，禁止将不同目的的改动混在同一个提交中。
- 提交前必须再次确认：本次提交内容与提交说明一一对应，且提交范围内不夹带无关文件或无关修改。
